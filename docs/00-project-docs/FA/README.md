# 可行性分析报告

## 1. 项目概述
### 1.1 项目名称
LingoBridge (对外中文学习平台 MVP)

### 1.2 项目背景
目前针对海外或留学生的中文教学，缺乏一个轻量、聚焦于“课件演示+跟读录音+作业打卡”闭环的教学工具。传统的 LMS 过于重型，通用的会议软件无法有效沉淀学习记录和细颗粒度的互动数据。

### 1.3 预期目标
打造一个极其轻量的、基于 Web 的语言学习平台。首期 MVP 提供：
1. 游客试用 / 学生、教师、管理员登录门禁。
2. 教师端上传课件（PDF/PPTX）、布置作业（Excel导入）、开启 Live 直播。
3. 学生端接入课堂，观看课件并进行跟读、提交练习记录。

## 2. 技术可行性分析
### 2.1 现有技术栈是否满足需求
本项目采用前端 React 19 + Vite 结合后端 Node.js (Express + tsx)，生态极其完善。
PDF.js 可实现前端原生渲染课件并支持白板标注，Web Audio API 支持学生端录音，技术实现完全可行。

### 2.2 关键技术难点识别
- **PDF 渲染与画笔的性能耦合**：课件翻页与画笔的结合可能导致重绘卡顿或白屏。通过解耦渲染层和交互层，以及提供清理特定页笔迹的功能来解决。
- **MVP 成本控制**：为避免过早引入复杂的基础设施，摒弃 MySQL/Redis，改用基于 Node.js `fs/promises` 的本地 JSON 数据持久化（`backend/data/db.json`），对 MVP 及本地验证来说极大地降低了门槛。

## 3. 经济可行性分析
### 3.1 开发成本估算
- 技术栈简单统一（全栈 TypeScript），单人全栈开发者或小型团队即可在短周期（1-2个 Sprint）内完成 MVP。
- 部署成本低，仅需一台普通 VPS 服务器跑 Docker（1核2G甚至足够）及对象存储成本（或本地存储）。

## 4. 操作可行性分析
### 4.1 目标用户群特征
- **学生**：需要极低门槛的跨平台访问体验，因此提供纯 Web 应用（无须下载安装客户端）。
- **教师**：日常教学中重度依赖 PDF 和 Excel 词表，故提供原生的 PDF/PPTX 上传和 Excel 题库解析。

## 5. 风险初步评估
| 风险项 | 可能性 | 影响度 | 应对策略 |
|---|---|---|---|
| JSON 读写在高并发下的数据丢失/冲突 | 中 | 高 | 现阶段控制试点班级人数，后期流量扩大前再迁移至 MySQL/PostgreSQL |
| PDF 在极低端设备上渲染内存溢出 | 中 | 中 | 对大文件提供转换并缓存的分页资源加载，避免一次性载入全量页面 |

## 6. 结论
[x] 可行，建议立项并以现有的 JSON + React 架构进入迭代闭环。
