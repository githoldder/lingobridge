## 1. 测试范围与目标

### 1.1 测试范围
当前版本集中力量覆盖 LingoBridge MVP 阶段的教学闭环，重点验证：
1. 各用户角色的权限与路由隔离逻辑。
2. 教师端上传资源（PDF、Excel）和开设 Live 课时的链路。
3. 学生端访问课程、渲染 PDF、录音上报。
4. 后端基于本地 JSON `db.json` 的数据持久化读写。

### 1.2 核心质量目标
- 学生/游客门禁100%覆盖：无法通过直接输入 URL 或篡改本地存储越权。
- 业务关键路径：Live创建 -> PDF/白板加载 -> 学生进入 -> 学习记录上传 必须无阻断性缺陷（P0）。

## 2. 测试策略与工具对齐

### 2.1 端到端测试 (E2E)
- **工具选型**：使用 **Playwright**。
- **运行命令**：`npm run e2e`。
- **重点场景**：
  - 打开浏览器使用 Teacher 账号登录，检查能否创建课程、上传 PDF。
  - 打开新的隐身窗口作为 Student，检查能否接受 Live 并进行麦克风授权。

### 2.2 后端接口/单元测试 (Backend API)
- **工具选型**：基于 Node.js 原生的 test runner 和 TSX 运行。
- **运行命令**：`npm run backend:test`
- **重点场景**：
  - 测试 `ensureDb()` 和 `writeDb()` 在极高频读写下的文件系统行为。
  - 测试 Excel 导入模块能否正确剔除乱码、提取出单词。

## 3. 测试进度与环境安排

| 阶段 | 重点内容 | 责任对象 | 对应环境 |
|---|---|---|---|
| **Smoke 测试** | 安装依赖后直接运行，通过 `npm run dev` 能否加载前端，`tsx backend/src/server.ts` 能否启动后端并自动建立 `db.json` 骨架。 | 开发者 (自测) | 本地 |
| **API/单元集成** | 通过 HTTP 请求确认所有的 `/api/v1/*` 不发生 500 错误。 | CI/CD 或自动化脚本 | 本地 / 测试服 |
| **功能回归** | 针对 PRD 中列出的 Sprint 缺陷（如画笔白屏、跳页不记录状态）进行重点测试复核。 | 测试/开发者 | 测试服 |
