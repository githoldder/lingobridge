\section{云基础设施及运行成本}
本系统在云端资源采购与持久运营层面的成本预算较为精简，各组件的配置规划如下：
\begin{itemize}
  \item \textbf{前端托管成本}：前端 UI 静态资源部署于 Vercel 边缘托管服务器，利用其提供的基本免费配额进行静态分发，可降低早期的多区域服务器租赁开销。
  \item \textbf{计算节点配置}：后端 API 容器部署于国内云主机上海地域的轻量级应用主机（Lighthouse 实例，硬件配置及带宽配置属于基础规格），年运行预算可控制在较低范围内。
  \item \textbf{数据库存储配置}：关系数据库在开发与小班测试阶段采用自建容器或云托管的 PostgreSQL，其在小并发连接下的冷启动和连接释放性能能够满足演示验证要求。
\end{itemize}

\section{ASR/TTS 费用优化与双层缓存机制}
由于口语学习需高频调用外部云厂商的 TTS（文本转语音）发音合成接口，发音合成的计费开销通常与请求字符数呈线性递增关系，需在设计上予以优化。

为了控制接口调用成本，项目组对主流智能语音服务商的免费额度与计费价格进行了审计\cite{Ref-SpeechAudit}。Azure 语音服务免费层（F0 级别）每月提供一定字符额度，但其免费并发请求受限制且中文字符存在特定的折算规则\cite{Ref-AzureTTS}。为此，项目组在后端设计了双层语音缓存策略（Double-Layer Audio Cache）：
\begin{itemize}
  \item \textbf{内存级 L1 缓存}：将高频词汇（如拼音字母）的合成音频流缓存在内存的映射对象中，以降低响应时延。
  \item \textbf{磁盘级 L2 缓存}：在后端持久化存储中设立缓存目录。当同一词汇或句子再次被教师或学生请求发音时，直接以静态资源形式返回，以避免重复调用外部接口。
\end{itemize}

如图 \ref{fig:cost_comparison} 所示，随着平台累积语音交互请求次数的递增，理论上由于缓存命中率不断逼近渐近线，使用双层缓存架构下的 API 调用累计总费用呈明显的对数收敛形态。双层语音缓存策略可有效降低云服务商 TTS 重复生成成本，具体节省比例取决于课程的复用率与固定词汇的使用率\cite{Ref-AzureTTS}。
