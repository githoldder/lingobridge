\chapter{技术可行性}

\section{现有技术栈评估}
LingoBridge 中文学习平台系统架构在设计之初，便致力于实现前期的轻量高敏捷开发与后期生产环境的级联硬化，其技术栈选型完全遵循国家软件生命周期规范\cite{gb8567}与软件需求规格规范\cite{gb9385}。本系统的核心技术栈选型及可行性评估如下：

在前端架构设计上，系统采用 React 19 配合 Vite 构建工具及 TypeScript 强类型语言。Vite 的热模块替换机制极大缩短了原型开发循环，编译后的静态单页面应用（SPA）资源仅约 1.5\,MB。此架构能够完美利用 Vercel 等全球边缘计算网络进行静态托管，显著降低跨国用户的首屏加载时延。

在后端信令层，系统采用轻量级的 Node.js 及 Express.js 框架。其非阻塞 I/O 和单线程事件循环机制天生适合高频的 Presence（在线状态）轮询和基于 WebRTC 的信令交换。

在数据存储架构方面，本平台开创性地设计了双模仓储（Repository）存储架构。为了规避早期原型开发过度依赖大型数据库进程的痛点，系统底层抽象了统一的数据访问接口。本地开发模式下默认启用单文件轻量级 JSON 存储（`readDb`/`writeDb` 机制），无需部署 any 数据库服务即可实现秒级冷启动；而在生产环境下，系统可无感一键切换至 PostgreSQL 关系数据库模式。该设计通过高度向下兼容的数据传输对象（DTO）映射逻辑，在保证开发效率的同时，完全硬化了生产模式下级联删除（CASCADE）与事务一致性。

\section{关键技术难点及对策}
在平台 0 到 1 的研发与生产部署生命周期中，项目组攻克了以下关键技术瓶颈：

针对大课件文件上传超时与 502 路由器限制的瓶颈，在海外 Vercel 静态托管平台与中国上海腾讯云轻量级主机（CVM）后端的协同部署下，网络跨海握手极易引发 Serverless Router 触发 of 502/504 错误。为此，项目组对 multipart/form-data 传输通道进行了多层解耦。当常规的流式分片上传在极限网络下超时，系统将自动触发 JSON-Base64 小包分块上传作为强制性兜底对策。

针对跨国低时延实时信令阻断的问题，受公网 HTTPS 路由阻断或 TLS 复杂握手时延影响，学生端在频繁进行课堂在线同步（Presence）与信令广播（Signals）时可能会发生心跳断连或重试阻塞。项目组对此在 CVM 宿主机上独立部署了容器化 Caddy 反向代理，绑定 `:80` 端口监听，并将 Vercel 前端代理指向 CVM 真实公网 IP。这不仅规避了复杂域名的 ISP HTTP 拦截，还利用 Caddy 极极速端口复用保障了 Presence 的平稳通信。同时，前端引入了 15 秒自适应弹性状态机（Adaptive Polling Loop）\cite{Ref-PlaywrightE2E}，在遭遇跨海长连接抖动时自动轮询规避控制台报错，成功率最终达 100\%。

在实时音视频集成可行性上，项目组通过对第三方会议系统 API（如 Microsoft Teams 与腾讯会议）的深入调研与对比评估\cite{Ref-TeamsTencent}，发现其企业开发者账号存在极高的准入门槛及漫长的审核周期。为保证 MVP 的轻量性，平台决定在初期放弃重型第三方视频 API 绑定，转而自主实现基于 WebRTC 的极简控制信道，极大地降低了集成风险。

针对学生多维录音打分持久化及 ASR 端侧评估，学生口语跟读练习需支持高频的 Web Audio API 麦克风录制，并在本地进行 Wav 帧合并与 ASR（自动语音识别）评估。系统在前端实现了 3-slot 自动覆盖跟读算法（每个跟读任务仅缓存并提交成绩最高的三次音轨），由后端根据 `taskId` 和 `lessonNodeId` 进行级联保存，有效防止了无用二进制数据占满硬盘，并通过评估端侧 3B 级别小模型在跨国降噪华文教学中的词错率（WER/CER）边界\cite{Ref-AsrPipeline}，确保了实时语音识别的可行性。
