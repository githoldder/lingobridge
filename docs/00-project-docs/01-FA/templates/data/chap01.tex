\chapter{技术可行性}

\section{现有技术栈评估与成熟度审计}
LingoBridge 中文学习平台系统架构在设计之初，便致力于在开发效率与生产环境稳定性之间取得平衡，其技术栈选型与设计参考了相关软件工程规范\cite{gb8567,gb9385}。本系统的核心技术栈选型及可行性评估如下：

在前端架构设计上，系统采用 React 19 配合 Vite 构建工具及 TypeScript 强类型语言。Vite 的热模块替换（HMR）机制有助于缩短开发与调试周期，提升开发响应效率。编译后的静态单页面应用（SPA）资源可通过全球边缘计算网络进行静态托管与分发，这极大地降低了跨国用户的首屏加载时延，避免了繁重的服务器端渲染（SSR）带宽消耗。

在后端信令层，系统采用轻量级的 Node.js 及 Express.js 框架。其非阻塞异步 I/O 和单线程事件循环机制非常适用于高频的在线状态（Presence）轮询与基于 WebRTC 的受控连麦信令交换。后端通过精简的信令管道逻辑，可在超低系统开销下承载留学生班级的同步翻页与状态广播行为。

在数据存储架构方面，系统抽象了统一的数据访问接口。本地开发模式下默认启用单文件轻量级 JSON 存储（基于 \texttt{readDb} 与 \texttt{writeDb} 机制），无需在开发初期部署和维护复杂的外部数据库服务即可实现冷启动，打消了对先期复杂环境依赖的顾虑；而在生产及正式试点环境下，系统设计了面向 PostgreSQL 关系数据库的抽象配置接口。该设计通过数据传输对象（DTO）映射逻辑与可插拔设计，在保证早期敏捷交付的同时，为后续向关系型多租户数据库过渡提供了良好的兼容性支持。

为了深入论证核心智能语音交互的可行性，项目组对主流智能语音服务商（包括腾讯云、阿里云以及 Azure 智能语音服务）在发音合成（TTS）与语音识别（ASR）维度进行了详细的技术与免费层额度审计\cite{Ref-SpeechAudit,Ref-AzureTTS}。审计对比分析如表 1.1 所示。

\section{关键技术难点及对策}
在平台原型研发与部署验证中，项目组针对跨国长距离通信面临的关键技术难点，设计了如下对策：

针对跨海网络上传课件面临的超时与网关限制瓶颈，由于海外静态托管平台与国内轻量级主机（CVM）后端的协同部署下，跨海网络连接存在引发 502 或 504 网关超时错误的风险。为此，项目组对文件传输通道进行了多层解耦。在当前的演示与测试阶段，系统主要通过限制上传课件页数（限制在 10 页以内）、压缩 PDF 文件体积（控制在 5MB 以下）以及预置演示课程数据来降低网络抖动引发的失败概率；而在长期技术路线中，规划采用对象存储直传（Presigned URL）或专有后端文件微服务进行应对。

针对跨国低时延实时信令的同步要求，公网 HTTPS 路由抖动或复杂握手时延可能导致心跳断连或重试阻塞。项目组在主机上部署了容器化反向代理（绑定 \texttt{:80} 端口监听），并将前端代理指向真实的公网 IP，以稳定 Presence 通信。同时，前端信令层引入了自适应指数退避与重连机制，在遭遇跨海长连接抖动时自动轮询重试，从而提升通信连接的成功率。

在实时音视频集成可行性上，经对 Microsoft Teams 与腾讯会议 API 的调研与评估\cite{Ref-TeamsTencent}，由于第三方会议系统的企业开发者账号存在准入门槛与审核周期限制，为保证 MVP 阶段的轻量性，平台在初期将以腾讯会议的 REST API 作为主要集成平台（提供入会链接生成与出勤数据拉取），同时保留 Microsoft Teams 作为海外学员的备选方案。实时音视频闭环与字幕翻译同步排除在 MVP 主线之外。

针对学生口语跟读与音频持久化需求，系统在前端基于 Web Audio API 实现麦克风录制，并在本地进行音频缓存与临时回放，由后端根据任务 ID 进行持久化存储。发音盲区自动高亮及 ASR（自动语音识别）口语打分功能目前列为后续扩展方向，规划在后续迭代中接入阿里云智能语音交互或腾讯云 ASR 的教育场景垂直模型，以利用其 95\% 以上的中文普通话识别准确率及流式延迟指标\cite{Ref-AsrPipeline}。
