\chapter{技术可行性}

\section{现有技术栈评估}
LingoBridge 中文学习平台系统架构在选型设计上，参考了相关软件工程规范以确保开发效率与生产环境稳定性的协调\cite{gb8567,gb9385}。本系统的核心技术栈选型及可行性评估如下：

在前端架构设计上，系统采用 React 19 配合 Vite 构建工具及 TypeScript 强类型语言。Vite 的热模块替换机制有助于缩短开发周期\cite{Ref-SpikePlan}。编译后的静态单页面应用（SPA）资源可通过全球边缘计算网络进行静态托管，以降低跨国用户的首屏加载时延（首屏加载时延指标优化至 500 毫秒以内）。

在后端信令层，系统采用轻量级的 Node.js 及 Express.js 框架。其非阻塞 I/O 和单线程事件循环机制适用于高频的在线状态（Presence，轮询间隔设定为 5 秒）轮询与基于 WebRTC 的信令交换。

在数据存储架构方面，系统抽象了统一的数据访问接口。本地开发模式下默认启用单文件轻量级 JSON 存储（基于 \texttt{readDb} 与 \texttt{writeDb} 机制）以在开发初期进行冷启动验证（该设计在 POC 验证中作为临时本地持久化方案，生产环境则提供 PostgreSQL 数据接口迁移规划）\cite{Ref-SpikePlan}；而在生产环境下，系统提供迁移至 PostgreSQL 关系数据库的配置接口，以确保数据持久化事务的一致性。

\section{关键技术难点及对策}
在平台原型研发与部署验证中，项目组针对关键技术难点设计了如下对策：

针对跨海网络上传课件面临的超时与网关限制瓶颈，在海外静态托管平台与国内轻量级主机（CVM）后端的协同部署下，跨海网络连接存在引发 502 或 504 网关超时错误的风险\cite{Ref-SpikePlan}。为此，项目组对文件传输通道进行了多层解耦。在当前的演示与测试阶段，系统主要通过限制上传课件页数（限制在 10 页以内）、压缩 PDF 文件体积（压缩至 5MB 以下）以及预置演示课程数据来降低网络抖动的失败概率；而在长期技术路线中，规划采用对象存储直传（Presigned URL）或专有后端文件微服务进行应对。

针对跨国低时延实时信令的同步要求，公网 HTTPS 路由抖动或复杂握手时延可能导致心跳断连或重试阻塞。项目组在主机上部署了容器化反向代理（绑定 \texttt{:80} 端口监听），并将前端代理指向真实的公网 IP，以稳定 Presence 通信。同时，前端信令层引入了自适应指数退避与重连机制，在遭遇跨海长连接抖动时自动轮询重试，从而提升通信连接的成功率。

在实时音视频集成可行性上，经对 Microsoft Teams 与腾讯会议 API 的调研与评估\cite{Ref-TeamsTencent}，由于第三方会议系统的企业开发者账号存在准入门槛与审核周期限制，为保证 MVP 阶段的轻量性，平台在初期将以腾讯会议的 REST API 作为主要集成平台（提供入会链接生成与出勤数据拉取），同时保留 Microsoft Teams 作为海外学员的备选方案。实时音视频闭环与字幕翻译同步排除在 MVP 主线之外。

针对学生口语跟读与音频持久化需求，系统在前端基于 Web Audio API 实现麦克风录制，并在本地进行音频缓存与临时回放，由后端根据任务 ID 进行持久化存储。发音评估与 ASR（自动语音识别）口语打分功能目前列为后续扩展方向，规划在后续迭代中优先接入 Azure Speech Services 的发音评估功能（Pronunciation Assessment）及流式 ASR 模型\cite{Ref-AsrPipeline,Ref-AzureTTS}，或根据实测情况引入火山引擎/百度智能云等国内 ASR 服务\cite{Ref-SpeechAudit}，以利用其 95\% 以上的中文普通话识别准确率指标。
