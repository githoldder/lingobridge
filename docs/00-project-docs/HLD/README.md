## 1. 引言

### 1.1 目的
定义 LingoBridge MVP 的高层系统架构、技术栈选型与模块划分，供前后端开发与部署提供全局视角的参考。

### 1.2 架构原则
- **极简主义**：避免过早引入复杂中间件和重型关系型数据库。
- **前后分离**：明确区分客户端渲染逻辑与服务端 API 及文件存储边界。

## 2. 技术架构设计

### 2.1 整体技术栈选型

| 层次 | 技术选型 | 选型理由 |
|---|---|---|
| **前端框架** | React 19 + Vite | 提供极速构建和成熟生态，利于快速迭代 |
| **前端样式** | TailwindCSS + Framer Motion | 高效编写响应式布局和实现流畅的课件/白板动画 |
| **课件渲染** | PDF.js | 原生支持浏览器内高性能的 PDF 渲染与分页控制 |
| **后端框架** | Node.js + Express (TSX运行) | 全栈 TypeScript 统一语言，TSX提供即时运行能力 |
| **数据存储** | 本地 JSON (`fs/promises`) | 极简持久化，MVP 阶段满足小规模封闭内测需求 |
| **部署托管** | PM2 / Docker + Nginx | 提供灵活的本地热重载开发与生产环境守护 |

### 2.2 系统分层

- **Web 客户端 (Frontend)**：处理 UI 渲染、`AuthContext` 权限派发、PDF 绘制与画笔工具逻辑、麦克风调用（Web Audio API）。
- **反向代理 (Nginx)**：承接 HTTP/HTTPS 请求，将前端页面作为静态资源伺服，或将 `/api` 和媒体文件代理至后端。
- **业务服务层 (Backend/Express)**：处理 JWT 鉴权、课程管理、Excel 数据解析、录音存储及数据校验。
- **数据访问层 (Local FS)**：通过自定义的 `backend/src/db.ts` 实现文件锁/并发安全的内存加载与 `db.json` 写入。

## 3. 核心模块划分

| 模块名 | 核心职责 | 前端/后端关注点 |
|---|---|---|
| **Auth 模块** | 登录注册、角色(Admin/Teacher/Student)分流 | 前端 GuestGate 门禁；后端 JWT 签发及校验 |
| **Course 模块** | 课程、Live Class、课时节点生命周期管理 | - |
| **Resource 模块**| 处理上传的 PDF/PPTX 和 Excel | 后端需将文件按班级目录隔离并更新 db.json |
| **Classroom 模块**| 课件同步、画笔渲染、白板控制 | 前端需解耦 canvas 绘制与 PDF 背景渲染 |
| **Admin 后台** | 数据看板，管理师生数据及错误记录 | 仅独立路由 `/admin` 可见，强制后端检查 Admin 角色 |

## 4. 关键设计决策 (ADR)

1. **为什么不用 MySQL/Redis？**
   - 现阶段重心在于跑通中文交互闭环，非高并发场景。单文件 JSON 通过全量拉取+增量写入已足够支撑数百名以内的试用。
2. **画笔为何不保存全量历史矢量而采用位图或按需清理？**
   - PDF 翻页与画板结合极其脆弱。当前方案为：关闭画笔不清空；清除画笔操作仅针对“当前页”状态，简化跨页回放的复杂状态机。
