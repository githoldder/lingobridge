# 引用与参考文献规范

> 本模块建立资料引用、证据链和参考文献管理规范。依据 GB/T 7714—2015《信息与文献 参考文献著录规则》以及国际通用引用标准。
>
> **核心原则：没有来源的内容不得进入正式文档；AI 生成内容不能作为事实来源。**

---

## 一、资料来源分级制度

### 1.1 五级分级体系

| 等级 | 标识 | 来源类型 | 可信度 | 在正文中的作用 |
|---|---|---|---|---|
| **L1 权威** | ⭐⭐⭐⭐⭐ | 国家标准（GB）、国际标准（ISO/IEC/IEEE）、法律法规、政府白皮书、最高人民法院司法解释 | 极高 | 可直接作为事实依据引用 |
| **L2 学术** | ⭐⭐⭐⭐ | 同行评审期刊论文（A类/B类）、学位论文（博士/硕士）、学术专著、有 DOI 的会议论文 | 高 | 可作为主要论据，需标注出处 |
| **L3 技术** | ⭐⭐⭐ | 知名开源项目官方文档（GitHub star ≥ 5000）、行业权威白皮书、有 ISBN 的技术书籍、知名厂商官方技术文档 | 中高 | 可作为技术方案的参考依据 |
| **L4 参考** | ⭐⭐ | 知名技术博客作者（有实名认证）、知乎高赞回答（有可溯源依据）、Stack Overflow 有来源的回答、CSDN/博客园有出处的技术文章 | 中 | 仅作辅助说明，不可作为主要依据 |
| **L5 不可靠** | ❌ | 匿名文章、无来源博客、停服/失效网页、社交媒体帖子、AI 生成内容、无 DOI 的普通中文期刊论文、内部文件（无编号） | 低/未知 | **严禁引用** |

### 1.2 每级判断标准（逐项核查）

对每条资料，按以下五项逐一评估：

| 评估项 | 通过条件 | 不通过时降级处理 |
|---|---|---|
| **① 明确的作者/机构** | 有可识别的真实作者或机构署名 | 无 → 降一级；匿名 → L5 |
| **② 可验证的来源** | 有 URL / DOI / ISBN / 标准号之一可查 | 无可验证标识 → 降一级 |
| **③ 原始来源** | 是原始发布，非转发/摘录/翻译 | 转发 → 追溯原始；找不到原始 → 降一级 |
| **④ 时效性** | 工程技术类 ≤ 5年；法规/标准查现行状态；学术论文 ≤ 10年（前沿领域 ≤ 3年） | 超期 → 标注"已过时"或换来源 |
| **⑤ 利益关联** | 无广告/PR/商业利益关联 | 广告/PR 内容 → 降一级且仅作参考 |

### 1.3 典型案例判断

| 资料 | 判断 | 理由 |
|---|---|---|
| 全国标准信息公共服务平台发布的 GB/T 8567—2006 | L1 | 政府网站 + 标准编号 + 可验证 |
| 中国知网收录的《计算机学报》论文（有 DOI） | L2 | 同行评审 + DOI 可验证 |
| GitHub 上 tensorflow/tensorflow 仓库 README | L3 | 知名开源 + 高 star + 官方维护 |
| 知乎用户"机器之心"的技术回答（引用了论文 DOI） | L4 | 有来源但经过转述，仅作辅助 |
| 某匿名博客"据知情人士透露……" | L5 | 匿名 + 无可验证来源 |
| ChatGPT 生成的代码解释 | L5 | AI 生成内容，非人类原创 |

## 二、资料元数据记录规范

### 2.1 必填字段（每条资料必须完整填写）

收集每条资料时，在文献管理工具（Zotero/EndNote 等）或 Obsidian 元数据表中填写以下全部字段：

```yaml
标题:                        # 文献/文章的完整标题，不得缩写
类型:                        # 期刊/图书/标准/网页/学位论文/专利/报告/API文档/开源项目
作者/机构:                   # 真实姓名或机构全称，多位作者如实列出
来源URL:                     # 可访问的稳定链接（临时链接需存档快照）
发布时间:                     # YYYY-MM-DD（不可确定时写"YYYY"或"[无]"）
访问日期:                     # YYYY-MM-DD（仅网页类必填，图书/论文不填）
内容摘要:                     # 50字以内，描述该资料的核心贡献
可信度评级:                   # L1 / L2 / L3 / L4（精确到等级）
适用场景:                     # 论文引用 / 技术方案 / 调研素材 / 辅助说明
关联主题:                     # 该资料关联的文档/章节/关键词
原始来源追溯:                 # 若为转发，填写原始来源URL；否则填"原创"
利益关联声明:                 # 有无广告/PR/商业赞助；无则填"无"
```

### 2.2 资料元数据表示例

```
标题: GB/T 7714—2015 信息与文献 参考文献著录规则
类型: 国家标准
作者/机构: 中华人民共和国国家质量监督检验检疫总局 中国国家标准化管理委员会
来源URL: https://openstd.samr.gov.cn/bzgk/std/newGbInfo?hcno=7FA63E9BBA56E60471AEDAEBDE44B14C
发布时间: 2015-05-15
访问日期: 2024-04-01
内容摘要: 规定了学术论文、学位论文、图书等各类文献的参考文献著录格式，是引用格式的基准规范
可信度评级: L1
适用场景: 论文引用（必依）
关联主题: 参考文献格式、引用规范
原始来源追溯: 原创
利益关联声明: 无
```

```
标题: Attention Is All You Need
类型: 期刊论文（会议论文扩展版）
作者/机构: Vaswani A, Shazeer N, Parmar N, et al.
来源URL: https://arxiv.org/abs/1706.03762
发布时间: 2017-06-12
访问日期: 2024-04-01
内容摘要: 提出 Transformer 架构，基于自注意力机制，在机器翻译任务上显著优于 RNN 模型
可信度评级: L2
适用场景: 论文引用 / 技术方案参考
关联主题: Transformer、自注意力机制、NLP
原始来源追溯: 原创（arXiv）
利益关联声明: 无
```

### 2.3 资料收集流程

```
Step 1: 发现资料 → 记录标题 + URL（立即）
Step 2: 初步评级（L1-L5）→ 判断是否值得深入阅读
Step 3: 阅读并填写完整元数据表
Step 4: 归类到对应主题/项目文件夹
Step 5: 网页类资料→立即存档快照（防止链接失效）
Step 6: 在文档中使用时标注引用编号
```

## 三、证据链记录规范

### 3.1 证据链的定义

每个进入正式文档的**重要结论**（即读者可能质疑或引用的结论），必须能够追溯到原始证据来源。证据链的逻辑结构为：

```
结论（Claim）
    ↓ 我为什么这么说？
证据（Evidence）—— 支持该结论的具体事实、数据或理论
    ↓ 这个证据从哪里来？
引用（Citation）—— 原始文献的完整著录信息
```

### 3.2 证据链三要素判断规则

| 要素 | 定义 | 常见错误 |
|---|---|---|
| **结论** | 你主张的核心观点，应简洁、可证伪 | ❌ 过于宽泛（如"AI很重要"）；❌ 多个结论混在一起 |
| **证据** | 支持结论的具体内容：实验数据/文献记载/逻辑推导/权威定义 | ❌ 证据与结论相关性弱；❌ 只有观点没有数据 |
| **引用** | 证据的原始来源，须精确到页码/章节/URL | ❌ 引用综述类文献而非原始文献；❌ 引用了但未核对原文 |

### 3.3 证据链记录表（必填）

为每个重要结论建立如下记录：

| 字段 | 填写说明 | 示例 |
|---|---|---|
| **结论编号** | 按章节顺序编号，如 `EC-01`、`EC-02` | EC-01 |
| **结论内容** | 完整的结论陈述句 | "Transformer 在机器翻译任务上的 BLEU 分数比 RNN 高出 4.6 分" |
| **结论所在章节** | 该结论最终出现的文档章节 | 第3章 3.2 节 |
| **证据类型** | 数据/文献/逻辑/权威定义/实验 | 数据 |
| **证据内容** | 具体的数字/事实/引文 | BLEU: Transformer 28.4 vs RNN 23.8（基于 WMT-14 英德翻译任务） |
| **引用来源** | 原始文献的标题/编号 | [7] Vaswani et al. 2017 |
| **引用格式** | 按 GB/T 7714—2015 格式化后的著录项 | Vaswani A, Shazeer N, Parmar N, et al. Attention Is All You Need[J]. NeurIPS, 2017: 5998-6008. |
| **可信度** | L1 / L2 / L3（证据来源的评级） | L2 |
| **备注** | 任何需要注意的问题 | 需核实原文页码 |

### 3.4 证据链审查规则

**每条证据链必须满足：**
- [ ] 结论有且仅有一个核心主张（不多于一句完整话）
- [ ] 证据与结论之间存在直接因果或充分支持关系
- [ ] 引用是原始来源（非二手引用，除非原始文献确实无法获取）
- [ ] 引用可以被读者独立查证（URL/DOI/页码均提供）
- [ ] 证据链在最终文档中有对应的文中标注

## 四、参考文献管理规则（全流程）

### 4.1 阶段一：收集

| 规则 | 说明 |
|---|---|
| **来源优先顺序** | 原创论文 > 权威综述 > 专著章节 > 技术文档 > 网页（尽量避免网页作为主要来源） |
| **即时记录元数据** | 打开文献时即在 Zotero/EndNote 中补充完整字段，不事后补录 |
| **网页快照存档** | 所有网页来源须在 Wayback Machine（web.archive.org）或本地保存快照 |
| **DOI 是黄金标准** | 有 DOI 的文献优先使用，DOI 不可更改、可跳转，是永久标识符 |
| **禁止收集 L5 来源** | 匿名博客、无来源帖等不得进入文献库 |

### 4.2 阶段二：整理

| 规则 | 说明 |
|---|---|
| **按主题/章节分类** | 不按文献类型分类，按文档写作时的引用场景分类 |
| **标注核心贡献** | 每篇文献用一句话说明"这篇文章解决了什么问题"，便于引用时快速定位 |
| **剔除重复引用** | 同一结论有多篇文献时，优先保留 L1/L2 级别；同级别保留最新或最权威的 |
| **追踪引用链** | 如果发现某篇重要文献引用了更早的原始研究，应追加记录原始文献 |
| **记录是否已读原文** | 仅读摘要的文献标注"待读全文"，正文引用前须读原文核实关键数据 |

### 4.3 阶段三：格式化

使用文献管理工具（Zotero/EndNote/Mendeley）生成 GB/T 7714—2015 格式参考文献列表，但**必须人工逐条核对**，工具输出≠完全正确。

**格式化重点核查项：**

| 核查项 | 正确示例 | 常见错误 |
|---|---|---|
| 中文作者姓名 | 李明, 王华 | Zhang San（不应写成拼音顺序） |
| 英文作者姓名 | Zhang S, Li H（姓全大写，名首字母） | Zhang San（名未缩写） |
| 期刊名 | 计算机学报（不缩写） | CJ / Computer Journal（缩写） |
| 出版年 | 2023 | 不写或写错 |
| 页码 | 234-250 | 只有起始页 |
| DOI | https://doi.org/10.1234/xxx | 写错或遗漏 |
| 同一文献多次引用 | 文后参考文献不重复出现相同条目 | 同一文献出现两条 |

### 4.4 阶段四：复核

| 复核步骤 | 操作 |
|---|---|
| **Step 1** | 正文中每个引用编号 `[X]` 与文后参考文献列表一一对应（无遗漏、无多余） |
| **Step 2** | 逐条阅读文后参考文献，核对是否与正文中引用内容一致 |
| **Step 3** | 确认所有引用来源均为 L1-L3（非 L4/L5） |
| **Step 4** | 确认无 AI 生成内容被作为引用来源 |
| **Step 5** | 检查时效性：技术文档引用年份距今是否 ≤ 5年；法规/标准是否仍为现行有效 |
| **Step 6** | 随机抽查 3-5 条引用，通过 DOI/URL 在原始数据库中核实条目信息准确性 |

## 五、GB/T 7714—2015 各类引用著录格式（含完整示例）

> 注：以下格式基于 GB/T 7714—2015 的顺序编码制（顺序编号制）。若学校/期刊要求著者-出版年制，格式会有差异，请以具体要求为准。

### 5.1 期刊文章（J）

**通用格式：**
```
[序号] 作者. 题名[J]. 刊名, 出版年, 卷(期): 起止页码.
```

**完整示例：**

```
[1] Vaswani A, Shazeer N, Parmar N, et al. Attention is all you need[J]. Advances in Neural Information Processing Systems, 2017, 30: 5998-6008.

[2] He K, Zhang X, Ren S, et al. Deep residual learning for image recognition[C]// Proceedings of the IEEE Conference on Computer Vision and Pattern Recognition. Las Vegas: IEEE, 2016: 770-778.

[3] 李明, 王华, 张伟. 面向对象软件测试方法研究[J]. 计算机学报, 2022, 38(5): 234-250.

[4] 刘强, 陈静. 基于深度学习的自然语言处理技术综述[J]. 软件学报, 2023, 34(7): 3015-3034. DOI: 10.13328/j.cnki.jos.006912.
```

**注意事项：**
- 作者超过 3 人时，写前 3 人加", 等."（中文）或 ", et al."（英文）
- 有 DOI 时建议标注 DOI，可不写 URL
- 英文刊名实词首字母大写，不用全部大写

### 5.2 图书（M]

**通用格式：**
```
[序号] 作者. 书名[M]. 出版地: 出版者, 出版年.
[序号] 作者. 书名[M]. 译者, 译. 出版地: 出版者, 出版年.
[序号] 作者. 书名[M]. 版本. 出版地: 出版者, 出版年.
```

**完整示例：**

```
[5] Pressman R S, Maxim B R. Software Engineering: A Practitioner's Approach[M]. 9th ed. New York: McGraw-Hill, 2020.

[6] 谭浩强. C程序设计[M]. 5版. 北京: 清华大学出版社, 2017.

[7] Sommerville I. Software Engineering[M]. 10th ed. Harlow: Pearson, 2015.

[8] 周志华. 机器学习[M]. 北京: 清华大学出版社, 2016.

[9] Gamma E, Helm R, Johnson R, et al. Design Patterns: Elements of Reusable Object-Oriented Software[M]. 王春晖, 译. 北京: 机械工业出版社, 2015.
```

**注意事项：**
- 出版地写城市名（如"北京"），不写国家
- 出版者尽量写全称（如"机械工业出版社"而非"机械工业出版社"）
- 版本写"第X版"或"Xth ed."

### 5.3 学位论文（D）

**通用格式：**
```
[序号] 作者. 题名[D]. 城市: 学位授予单位, 年份.
```

**完整示例：**

```
[10] 张伟. 基于深度学习的图像识别算法研究[D]. 北京: 清华大学, 2021.

[11] Chen L. Machine learning applications in healthcare: a systematic review[D]. Cambridge: University of Cambridge, 2022.
```

**注意事项：**
- 城市写学位授予单位所在城市
- 博士/硕士学位均可使用此格式
- 不写导师姓名（GB/T 7714—2015 未要求）

### 5.4 国家标准（S）

**通用格式：**
```
[序号] 发布机构. 标准编号 标准名称[S]. 发布日期.
```

**完整示例：**

```
[12] 中华人民共和国国家质量监督检验检疫总局, 中国国家标准化管理委员会. GB/T 7714—2015 信息与文献 参考文献著录规则[S]. 2015-05-15.

[13] 中华人民共和国国家质量监督检验检疫总局, 中国国家标准化管理委员会. GB/T 8567—2006 计算机软件文档编制规范[S]. 2006-07-27.

[14] 国家市场监督管理总局, 国家标准化管理委员会. GB/T 7713.1—2025 信息与文献 编写规则 第1部分：学位论文[S]. 2025-08-01.
```

**注意事项：**
- 发布日期写完整日期（YYYY-MM-DD）
- 标准名称写全称，不得缩写
- 可在 URL 前加"来源："说明获取平台

### 5.5 国际标准（S）

**通用格式：**
```
[序号] 发布机构. 标准编号 标准名称[S]. 发布年.
```

**完整示例：**

```
[15] IEEE Computer Society. IEEE 830-1998 Recommended Practice for Software Requirements Specifications[S]. 1998.

[16] ISO, IEC. ISO/IEC/IEEE 12207:2017 Systems and Software Engineering — Software Life Cycle Processes[S]. 2017.

[17] IEEE. IEEE 1016-2009 IEEE Standard for Design Description[S]. 2009.
```

**注意事项：**
- ISO/IEC/IEEE 联合发布的标准可简写为"ISO/IEC/IEEE"
- 发布机构缩写须与标准号匹配

### 5.6 论文集/会议论文（C）

**通用格式：**
```
[序号] 作者. 题名[C]// 论文集名称. 出版地: 出版者, 出版年: 起止页码.
```

**完整示例：**

```
[18] Wang L, Zhang Y. Machine learning in medical imaging[C]// Proceedings of International Conference on Bioinformatics. Beijing: Springer, 2022: 156-163.

[19] LeCun Y, Bottou L, Bengio Y, et al. Gradient-based learning applied to document recognition[C]// Proceedings of the IEEE. New York: IEEE, 1998: 2278-2324.
```

### 5.7 专利（P）

**通用格式：**
```
[序号] 专利权人. 专利名称: 专利号[P]. 申请日期或授权公告日期.
```

**完整示例：**

```
[20] 华为技术有限公司. 一种数据处理方法及装置: CN202110456789.0[P]. 2021-04-23.

[21] Google LLC. Method and system for attention-based sequence modeling: US10861102[P]. 2020-08-04.
```

### 5.8 报告/技术白皮书（R]

**通用格式：**
```
[序号] 作者/机构. 题名[R]. 出版地: 出版机构, 年份.
```

**完整示例：**

```
[22] 中国信息通信研究院. 中国人工智能产业发展指数报告[R]. 北京: 中国信息通信研究院, 2023.

[23] McKinsey & Company. The State of AI in 2023: Generative AI's Breakout Year[R]. New York: McKinsey & Company, 2023.
```

### 5.9 电子资源（EB/OL）

**通用格式：**
```
[序号] 作者. 题名[EB/OL]. (更新日期)[访问日期]. URL.
```

**完整示例：**

```
[24] 中华人民共和国国家标准全文公开系统. GB/T 7714—2015 信息与文献 参考文献著录规则[EB/OL]. (2023-01-01)[2024-04-01]. https://openstd.samr.gov.cn/bzgk/std/newGbInfo?hcno=7FA63E9BBA56E60471AEDAEBDE44B14C.

[25] OpenAI. ChatGPT API Documentation[EB/OL]. (2024-03-01)[2024-04-01]. https://platform.openai.com/docs.

[26] 全国标准信息公共服务平台. GB/T 8567—2006 计算机软件文档编制规范[EB/OL]. [2024-04-01]. https://std.samr.gov.cn/search/std?keywords=GB%2FT+8567.
```

**注意事项：**
- 作者为机构时，直接写机构名称（无需"来源："前缀）
- 无更新日期时写发布日期；两者都没有时写"[无]"
- 访问日期必须标注

## 六、网页引用规则（分场景）

### 6.1 政府/官方机构网站

| 来源示例 | 引用格式要点 | 示例 |
|---|---|---|
| 政府官网（gov.cn） | 机构名 + 文档标题 + [EB/OL] + (更新日期/发布日期) + [访问日期] + URL | 中华人民共和国中央人民政府. 《"十四五"数字经济发展规划》解读[EB/OL]. (2022-01-12)[2024-04-01]. http://www.gov.cn/zhengce/2022-01/12/content_5668243.htm. |
| 全国标准信息公共服务平台 | 机构名 + 标准编号+标准名称 + [EB/OL] + [访问日期] + URL | 全国标准信息公共服务平台. GB/T 8567—2006 计算机软件文档编制规范[S]. [2024-04-01]. https://std.samr.gov.cn/search/std?keywords=GB%2FT+8567. |

### 6.2 新闻媒体报道

| 规则 | 说明 |
|---|---|
| 优先使用权威媒体 | 新华社、人民日报、人民网、央视新闻等 |
| 引用新闻中的数据/事实需交叉验证 | 同一事实有多源印证方可使用 |
| 新闻报道不可作为学术观点的唯一来源 | 须追溯到原始研究/权威报告 |

**示例：**
```
[27] 新华社. 我国人工智能产业规模突破5000亿元[EB/OL]. (2024-03-20)[2024-04-01]. http://www.news.cn/20240320/xxx.htm.
```

### 6.3 技术博客（个人作者）

**条件：可引用当且仅当满足以下全部条件**

- 作者实名（非匿名）
- 有明确发布/更新时间
- 内容有原始来源（引用了论文/文档/数据，而非凭空撰写）
- 非广告/PR 内容

**格式：**
```
[序号] 作者. 题名[EB/OL]. (发布日期)[访问日期]. URL.
作者: 李明（独立开发者）. 基于React的后端架构实践[EB/OL]. (2023-08-15)[2024-04-01]. https://liming.dev/react-backend-arch/.
```

### 6.4 知乎/问答类网站

| 规则 | 说明 |
|---|---|
| 知乎回答可引用**当且仅当** | 回答中有明确的数据来源或引用了论文/文档 |
| 纯个人经验分享（无来源） | **不可引用**，仅作参考 |
| 知乎专栏文章 | 视同技术博客处理，需满足实名+有来源条件 |

**示例（可引用）：**
```
[28] 张很高兴. 【论文解读】Transformer 的核心思想与实验结果[EB/OL]. (2022-11-05)[2024-04-01]. https://zhuanlan.zhihu.com/p/589351234. （注：文中引用了 Vaswani et al. 2017 原文）
```

**示例（不可引用）：**
```
❌ 匿名用户. 个人观点：我认为 ChatGPT 会改变所有行业。（无来源，不可引用）
```

### 6.5 社交媒体（微博/X/微信）

**禁止引用**。原因：非正式来源、内容可删除、无法验证、并非原始发布。

如确需引用数据，查找该信息的原始权威来源（政府报告/学术论文/官方公告）。

### 6.6 Wiki 类网站（维基百科等）

**谨慎使用**：维基百科本身不是可靠来源，但可以指引到原始文献。

规则：引用维基百科的内容，而非引用维基百科本身（即去维基百科查看参考文献，再引用原始文献）。

## 七、标准引用规则

### 7.1 国家标准（GB 系列）

**引用格式：**
```
[序号] 发布机构. 标准编号 标准名称[S]. 发布日期.
```

**引用场景与格式差异：**

| 场景 | 格式 | 示例 |
|---|---|---|
| 正文中首次提及 | "根据 GB/T 7714—2015《信息与文献 参考文献著录规则》" | — |
| 文后参考文献列表 | 完整著录格式 | [12] 中华人民共和国国家质量监督检验检疫总局, 中国国家标准化管理委员会. GB/T 7714—2015 信息与文献 参考文献著录规则[S]. 2015-05-15. |

**查询渠道：**
- 全国标准信息公共服务平台：https://std.samr.gov.cn（免费公开）
- 国家标准全文公开系统：https://openstd.samr.gov.cn（部分免费）
- 实施状态须在上线前核实（标准可能被废止/替代）

### 7.2 行业标准（HB/QB/YB 等）

**引用格式：**
```
[序号] 发布机构. 标准编号 标准名称[S]. 发布日期.
```

**示例：**
```
[29] 中国电子技术标准化研究院. SJ/T 11364—2017 电子信息产品有毒有害物质的标识[S]. 2017-01-19.
```

### 7.3 国际标准（ISO/IEC/IEEE/ANSI 等）

**引用格式：**
```
[序号] 发布机构. 标准编号 标准名称[S]. 发布年.
```

**常用机构缩写对照：**

| 缩写 | 全称 |
|---|---|
| ISO | International Organization for Standardization |
| IEC | International Electrotechnical Commission |
| IEEE | Institute of Electrical and Electronics Engineers |
| ANSI | American National Standards Institute |
| NIST | National Institute of Standards and Technology（美国） |

**示例：**
```
[30] ISO. ISO 9001:2015 Quality Management Systems — Requirements[S]. 2015.

[31] IEEE Computer Society. IEEE Std 829-2008 IEEE Standard for Software and System Test Documentation[S]. 2008.
```

### 7.4 被代替标准的引用规则

当一个标准被新版标准全部代替时，引用格式为：

```
[序号] 发布机构. 原标准编号 标准名称[S]. 发布日期. （被 GB/T XXXX—XXXX 代替）
```

在正文中说明："依据 GB/T 8567—2006（现行有效）"而非使用已废止版本。

## 八、开源项目引用规则

### 8.1 引用原则

1. **优先引用官方文档**（README/官方文档网站），而非第三方博客的解读
2. **优先引用特定版本/提交**，而非 always 最新版（防止链接内容漂移）
3. **GitHub/GitLab 仓库链接**须附上具体版本 tag 或 commit hash
4. **代码类引用**须同时标注许可证类型（如 Apache-2.0 / MIT）

### 8.2 GitHub 项目引用格式

**通用格式：**
```
[序号] 作者/机构. 项目名称（仓库描述）[EB/OL]. 仓库URL. 发布年份/版本/提交hash.
```

**完整示例：**

```
[32] TensorFlow Team. TensorFlow: Large-Scale Machine Learning on Heterogeneous Distributed Systems[EB/OL]. (2024-03-15)[2024-04-01]. https://github.com/tensorflow/tensorflow. License: Apache-2.0.

[33] van Rossum G. Python Programming Language[EB/OL]. (2023-10-01)[2024-04-01]. https://github.com/python/cpython. Version: 3.12.0.

[34] Contributors. React: A JavaScript library for building user interfaces[EB/OL]. (2024-02-28)[2024-04-01]. https://github.com/facebook/react. License: MIT.
```

### 8.3 开源项目文档引用

若引用的是开源项目的官方文档（非仓库本身），格式为：

```
[序号] 项目名称. 文档标题[EB/OL]. 文档URL. (版本/发布日期)[访问日期].
```

**示例：**
```
[35] Kubernetes Authors. Kubernetes Documentation: Workload Management[EB/OL]. (2024-01-01)[2024-04-01]. https://kubernetes.io/docs/concepts/workloads/.
```

### 8.4 开源许可证引用

在引用代码时，应注明许可证：

```
[36] Contributors. Vue.js (MIT License)[EB/OL]. https://github.com/vuejs/vue. 2024. License: MIT.
```

## 九、AI 辅助写作记录规范

### 9.1 AI 使用场景分级

| 级别 | 使用场景 | 是否需要记录 | 记录方式 |
|---|---|---|---|
| **A 类（可接受）** | 润色语言表达、纠正语法错误、生成代码框架示例、总结非核心参考资料（仍需人工核实） | 需要记录工具名称和使用场景 | 在附录 AI 使用声明中说明 |
| **B 类（需谨慎）** | 辅助整理调研资料、生成初稿框架（需大量人工修改） | 需要完整记录 | 在附录 + 脚注双重说明 |
| **C 类（禁止）** | 将 AI 生成内容直接作为事实来源、将 AI 内容作为引用来源、未经修改直接提交 AI 生成的整段文字 | — | **严禁** |

### 9.2 AI 使用声明格式（强制）

在文档末尾或摘要页后添加 AI 使用声明节：

```markdown
## AI 使用声明

本文在写作过程中使用了以下 AI 工具辅助：
| 工具名称 | 版本 | 使用场景 | 使用阶段 |
|---|---|---|---|
| ChatGPT (OpenAI) | GPT-4 | 英文摘要语言润色 | 定稿阶段 |
| 文心一言 (百度) | 4.0 | 技术术语中英对照参考 | 调研阶段 |

本文所有事实性结论、数据和引用均来自原始文献（见参考文献列表）。
AI 工具未作为任何事实主张的信息来源。
AI 辅助产生的非事实内容（如示例代码、框架草稿）均已经过人工审核和修改。
```

### 9.3 正文中的 AI 使用标注

当某段文字主要由 AI 辅助生成时，在该段落脚注或段末标注：

```
[^1]: 本段由 ChatGPT (OpenAI, GPT-4) 辅助语言表达，内容经作者核实。
```

### 9.4 AI 生成内容的限制

**绝对禁止：**
- 以任何形式暗示 AI 生成内容是"研究结果"或"实验数据"
- 在参考文献中出现 AI 模型名称（如"ChatGPT, 2024"）
- 将 AI 对话输出作为可引用来源

**如果 AI 提供了某个知识点：**
1. 在原模型对话中追溯 AI 提到的参考来源（AI 可能虚构，需核实）
2. 找到并引用人类原创的原始文献
3. 在文档中注明"经 AI 辅助定位，原始来源为 [X]"

### 9.5 防止 AI 编造引用的核验流程

当 AI 在写作中提供了引用信息时，按以下流程核验：

```
Step 1: 记录 AI 给出的"引用"信息（作者/标题/年份/期刊）
Step 2: 在原始数据库中独立搜索验证（Google Scholar / CNKI / IEEE Xplore）
Step 3: 若找不到对应文献 → 标记为"疑似虚构引用"，不使用
Step 4: 若找到 → 核对原文，确认文献内容与 AI 描述一致
Step 5: 确认一致后，以正规格式引用
```

## 十、不可引用资料清单

### 10.1 全面清单（禁止引用类型）

| 类别 | 具体情形 | 原因 | 替代方案 |
|---|---|---|---|
| **匿名来源** | 匿名博客/论坛/帖子 | 无法追溯，无人负责 | 引用原始权威来源 |
| **无来源内容** | "据知情人士透露"、微信截图（无出处） | 无法验证 | 查找正式公开资料 |
| **停服/失效网页** | 链接已不可访问 | 无法核实内容 | 使用 Wayback Machine 存档；或更换来源 |
| **社交媒体帖文** | 微博/推文/X/小红书帖子 | 非正式来源，无同行评审 | 查找同期权威报道 |
| **AI 生成内容** | ChatGPT/文心一言/Kimi 等生成内容 | 非人类原创，无原创性 | 追溯 AI 引用的原始人类文献 |
| **广告/PR 内容** | 企业软文、产品宣传稿 | 利益相关，不客观 | 找中立来源或一手数据 |
| **内部文件（无编号）** | 公司内部邮件、流程文档（无公开编号） | 不可公开验证 | 引用正式公开版本或行业通用做法 |
| **停刊/停更博客** | 内容超过 5 年未更新且无后续维护 | 信息可能过时且无后续修订 | 查找替代的权威来源 |
| **无 DOI 的普通中文期刊论文** | 无 DOI、无核心期刊标识、质量参差不齐 | 质量无法保证 | 优先使用核心期刊/A 类期刊（有 DOI） |
| **二手转引（无法溯源）** | 引用了某篇文献，但该文献引用了另一篇文献（且无法获取原始文献） | 可能存在信息传递错误 | 尽量找到原始文献；如无法找到，在文中说明"转引自[来源]" |
| **翻译/转载文章（无原出处）** | 仅在非正式平台看到某篇文章，无法找到原始出处 | 无法验证准确性 | 找到原始语言的文献 |
| **个人经验分享（无数据支撑）** | 知乎/博客中"我觉得……""经验是……"且无任何数据/文献支撑 | 主观性过强，无可验证性 | 找有数据支撑的来源 |
| **过时法规/废止标准** | 引用了已被替代/废止的法规或标准版本 | 内容已无法律效力 | 引用现行有效版本 |

### 10.2 降级使用规则

对于 L4 级别的资料，如必须使用，须满足：

1. 在正文明确标注："据 [来源] 报道"而非"根据 [来源] 研究"
2. 同时提供更高等级来源作为佐证
3. 不得作为主要论据，仅作辅助说明

## 十一、引用审查清单

### 11.1 引用前审查（写作时逐条核对）

**来源可信度：**
- [ ] 该引用来源的评级是 L1 / L2 / L3 吗？（不含 L4/L5）
- [ ] 该引用是原始来源（非转发）吗？
- [ ] 该引用是否支持你的结论（相关性足够强，而非仅"相关但不能证明"）？

**时效性：**
- [ ] 工程技术类引用：年份距今是否 ≤ 5年？
- [ ] 学术论文引用：是否在合理时间窗口（前沿领域 ≤ 3年，其他 ≤ 10年）？
- [ ] 法规/标准引用：是否为现行有效版本？（在发布前通过 https://std.samr.gov.cn 核实）

**可验证性：**
- [ ] 有 DOI / URL / ISBN / 标准号可独立验证吗？
- [ ] 读者能通过该信息独立找到原始来源吗？

**完整性：**
- [ ] 每条引用都填写了元数据表吗？
- [ ] 每条引用都建立了证据链记录吗？

### 11.2 引用格式审查（完稿后系统性检查）

**正文标注：**
- [ ] 正文中每个 `[X]` 引用编号在文后参考文献中都有对应条目吗？
- [ ] 引用编号顺序正确（按首次出现顺序编号）吗？
- [ ] 采用著者-出版年制时，作者姓名和年份是否正确？

**文后参考文献：**
- [ ] 所有文后参考文献在正文中均被引用过（无孤儿条目）吗？
- [ ] 参考文献条目的文献类型标识正确（[J]/[M]/[D]/[S]/[EB/OL] 等）吗？
- [ ] 中文文献的作者姓名格式正确（姓前名后）吗？
- [ ] 英文文献作者姓全大写，名首字母大写（如"Zhang S"而非"Zhang San"）吗？
- [ ] 期刊名写全称，不缩写吗？
- [ ] 期刊文章有起止页码吗？
- [ ] 有 DOI 的条目标注了 DOI 吗？
- [ ] 网页条目有访问日期吗？

**GB/T 7714—2015 格式专项：**
- [ ] 各类型格式完全符合 GB/T 7714—2015 规定？（参见第五章著录格式）
- [ ] 使用了文献管理工具（Zotero/EndNote）自动格式化后逐条人工核对了吗？

### 11.3 证据链审查（终稿前）

- [ ] 每个重要结论（涉及数据/事实/他人研究成果的结论）都有对应引用吗？
- [ ] 没有以 AI 生成内容作为事实来源吗？
- [ ] 没有以社交媒体/匿名博客作为主要论据吗？
- [ ] 所有引用都追溯到了人类原创的一手来源吗？
- [ ] 转引是否注明了"转引自"，且确有无法获取原始文献的理由？

### 11.4 引用质量审查（提交前）

- [ ] 引用数量充足：学术论文通常 ≥ 30 篇（因学科而异），课程论文通常 ≥ 10 篇
- [ ] 近年文献占比合理：通常近 5 年文献 ≥ 40%（技术文档可适当放宽）
- [ ] 核心观点有原始文献支撑（不依赖综述类文献作为唯一来源）
- [ ] 外文文献占比合理（通常 ≥ 30%，因学科要求而异）
- [ ] 自引比例合理（不超过总引用数的 20%，特殊情况除外）

## 附录 B：常见错误示例与修正

| 错误写法 | 正确写法 | 说明 |
|---|---|---|
| `[1] Zhang San, Li Hua. Title[J]. ...` | `[1] Zhang S, Li H. Title[J]. ...` | 英文作者名应缩写，只保留首字母 |
| `[1] 计算机学报, 2023(5): 12.` | `[1] 作者. 题名[J]. 计算机学报, 2023, 45(5): 12-25.` | 缺少作者、题名、页码范围 |
| `[1] https://example.com`（仅写 URL） | `[1] 作者. 题名[EB/OL]. (日期)[访问日期]. URL.` | 必须有完整的著录格式 |
