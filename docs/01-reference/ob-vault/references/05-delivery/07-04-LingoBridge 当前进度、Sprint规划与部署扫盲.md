## 二、眼前最重要的目标

老师真正关心的是四句话：

1. **能不能公网打开？**  
   要给一个能访问的网址或公网 IP。

2. **老师和学生怎么用？**  
   要有老师版、学生版操作手册。

3. **能不能组织学生采集数据？**  
   要有采集流程、录音标准、问题反馈表。

4. **需要多少钱？**  
   要有当前试用预算和后续正式运行预算。

所以接下来的 Sprint 不以“继续调研更多技术”为主，而以“演示闭环稳定、公网可访问、材料可交付”为主。

## 三、老师提出的“0 成本技术方案”如何理解

老师提出的方案如下：

| 组件 | 工具 | 费用口径 |
|---|---|---:|
| 前端 | Cloudflare Pages + Astro | 0 |
| AI 对话 | 智谱 GLM-4-Flash | 免费/低成本 |
| 语音合成 | Edge TTS | 免费但非正式服务风险 |
| 语音识别 | 浏览器 Web Speech API | 0 |
| 汉字动画 | Hanzi Writer 开源库 | 0 |
| 数据 | HSK 开源词汇数据集 | 0 |
| 用户系统 | Supabase 免费版 | 0 |
| 部署 | Cloudflare Pages | 0 |
| 通知推送 | Telegram Bot API | 0 |

这个方案的价值在于：它提醒我们要把 MVP 成本压低，优先使用免费层、开源库和浏览器能力。这个方向是对的。

但从当前 LingoBridge 的实际开发状态看，它不适合整套替换现有系统。当前项目已经是：

```text
React/Vite 前端
+ Express API 后端
+ Postgres schema/repository 数据层
+ PDF.js 课件渲染
+ XLSX 作业导入
+ TTS provider facade
+ 录音/课件/作业/Live/Admin 业务接口
```

现在最关键的问题不是“用 Astro 还是 React”，而是：老师创建的课程、课件、作业和 Live Class，学生能不能通过真实账号和真实课程关系看到并完成任务。

### 3.1 哪些可以直接吸收

| 技术 | 是否建议吸收 | 用法 |
|---|---|---|
| Cloudflare Pages | 可以 | 可用于部署当前 Vite 前端，不必改成 Astro |
| 智谱 GLM-4-Flash | 可以后续接入 | 作为 AI 对话 provider，不影响主链路 |
| Web Speech API | 可以作为辅助 | 做浏览器端语音识别/练习尝试，但不能作为唯一可靠 ASR |
| Hanzi Writer | 推荐 | 后续加入汉字笔顺动画和练习模块 |
| HSK 开源词汇 | 推荐 | 作为 seed 词库或 demo 数据，不能替代老师 Excel 作业 |
| Telegram Bot API | 可后置 | 用于通知老师/管理员，不进入当前 Sprint 主线 |

### 3.2 哪些不建议现在替换

| 技术 | 不建议原因 | 当前策略 |
|---|---|---|
| Astro 替换 React/Vite | 会重写前端，收益低，风险高 | 保留 React/Vite，必要时只改部署平台 |
| Supabase 替换现有用户/数据库层 | 当前已设计 Postgres schema 和 repository，切换会打断 Sprint 4 | 可把 Supabase 当托管 Postgres/Auth 的备选，不现在改主架构 |
| Edge TTS 作为主 TTS | 常见 Edge TTS 是非官方在线服务，稳定性和合规性不可控 | 保留当前 Azure Speech + browser fallback 架构，可做 demo fallback |
| 只用 Cloudflare Pages 承担全部后端 | Pages 更适合前端和轻函数，课件/录音/Excel/Live/Admin 后端逻辑较重 | 前端可上 Pages，后端仍独立部署或后续再评估 Workers |

### 3.3 给老师的简明解释

可以这样向老师说明：

> 老师提出的 0 成本方案方向是对的，里面的 Cloudflare Pages、Hanzi Writer、HSK 词库、Web Speech API、Telegram 通知都可以逐步吸收。但当前 LingoBridge 已经有 React 前端、Express 后端、Postgres 数据模型、PDF 课件和 Excel 作业导入链路。现在如果整套换成 Astro + Supabase，会打断正在收口的课程、学生、课件、作业和 Live Class 主流程。因此短期策略是：不推倒重做，而是在现有架构上吸收免费工具，先把 MVP 演示和数据采集闭环跑通。

## 四、当前开发真实状态同步

截至当前开发版本，`prd.json/prd.md` 已经把 Sprint 4 拆成可执行任务。状态口径如下：

| 模块 | 当前状态 | 说明 |
|---|---|---|
| Postgres schema | 基本完成 | 已有用户、课程、课时、作业、文件、Live、学习记录等表设计 |
| Repository 数据层 | 基本完成但有 follow-up | users/courses/assignments/live/files 等 repository 已建立，部分 handler 仍在迁移 |
| 登录注册 | 已接入真实数据层但需验收 | 仍需确认 production 密码策略、session/token 策略 |
| 教师-学生关系 | 已有基础能力 | 支持 teacher_student_links、course_members、学生搜索等方向 |
| 教师课程编辑页 | 下一步重点 | 需要把课程信息、学生、课件、作业、live 时间统一收口 |
| 学生端真实数据 | 下一步重点 | 要彻底移除静态 schedule 和 course-1 fallback |
| Live 翻页 | 下一步重点 | 要让 `liveSession.currentPage` 成为唯一可信页码 |
| PDF 批注 | 下一步重点 | 要拆分 PDF canvas 和 annotation layer，避免画笔导致白屏 |
| Excel 作业 | 待增强 | Excel 导入已存在，后续要补导入预览和手动表单模式 |
| Admin 后台 | 已有基础页面 | 需要补 admin 顶栏、logout、真实资源/进度核查和三角色 E2E |
| PPTX 策略 | 后置 | 本轮只保留上传/提示，不做 PPTX 原生预览或 Microsoft 365 Embed |

当前优先级不是继续换技术栈，而是按下面顺序把闭环打通：

1. 教师课程编辑闭环：课程信息、学生、课件、作业、Live 时间。
2. 学生真实数据入口：学生只看到自己加入的课程、日程、Live 和作业。
3. Live 翻页唯一可信：教师翻页写 live state，学生跟随。
4. PDF 批注稳定：PDF 渲染层和画笔/高亮/文字批注层分离。
5. Admin + 三角色验收：admin 退出、老师创建课程、学生看到课程并完成作业。

## 七、公网部署扫盲

### 1. 什么是服务器？

服务器就是一台 24 小时在线的远程电脑。我们把 LingoBridge 放到这台电脑上，老师和学生就可以通过网址访问。

### 2. 什么是公网 IP？

公网 IP 是服务器在互联网上的地址。没有域名时，可以直接用：

```text
http://服务器公网IP/
```

优点是立刻能用；缺点是不够好记。

### 3. 什么是域名？

域名是更容易记的网址，例如：

```text
https://lingobridge-demo.com
```

域名需要购买并解析到服务器公网 IP。

### 4. 什么是 HTTPS？

HTTPS 是加密访问。正式给学生使用时建议配置 HTTPS，因为浏览器对麦克风、摄像头、录屏等能力在安全环境下支持更稳定。

### 5. 什么是安全组？

安全组是云服务器的防火墙。即使程序启动了，如果安全组没有开放端口，外网仍然打不开。

常见端口：

| 端口 | 用途 |
|---:|---|
| 80 | HTTP 网站访问 |
| 443 | HTTPS 网站访问 |
| 3001 | 后端 API 测试端口 |
| 4174 | Vite preview 测试端口 |

正式部署建议用 80/443 对外，后端 API 由 Nginx 转发，不直接暴露太多端口。

### 6. 什么是前端和后端？

| 名称 | 通俗解释 | 本项目例子 |
|---|---|---|
| 前端 | 老师和学生看到的网页界面 | 登录页、课程页、录音按钮 |
| 后端 | 处理数据和文件的服务 | 登录、上传、录音保存 |
| 数据库/数据文件 | 保存用户、课程、记录 | Sprint 4 已建立 Postgres schema/repository，JSON 仅作过渡兼容 |
| 存储 | 保存课件、录音、录播文件 | 当前 MVP 用服务器目录 |

### 7. 什么是 PM2？

PM2 是 Node.js 服务守护工具。服务器重启或程序崩溃后，它可以帮助服务重新拉起。MVP 部署用 PM2 比较轻量。

### 8. 什么是 Docker？

Docker 是把应用和运行环境打包的工具。正式部署更稳，但配置成本更高。当前可以先用 PM2 快速上线，后续再 Docker 化。
