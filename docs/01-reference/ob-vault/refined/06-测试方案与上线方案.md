# LingoBridge 测试方案与上线部署规范

## 1. 测试用例矩阵 (TC-001 ~ TC-009)

### 1.1 功能测试用例

#### TC-001：教师端 PPT/PDF 课件解析导入
- **用例目的**：验证课件成功导入并解析为高保真图片课程页的能力。
- **前置条件**：教师账号 `teacher@test.com` 已登录控制台。
- **测试步骤**：
  1. 进入课件管理后台，点击“上传课件”；
  2. 选择 12MB 的 `.pptx` 文件并提交；
  3. 观察课件解析转换排队状态，等待其变为 `completed`。
- **预期结果**：后端 LibreOffice 解析完好率 **≥95%**，课程列表中生成包含页数正确的课程卡片。
- **验收标准**：A1（上传解析成功率） | 优先级：P0

#### TC-002：学生端翻页学习性能
- **用例目的**：验证多端响应式翻页的顺畅度。
- **测试步骤**：进入选定课程学习页，使用滑动手势或点击翻页按钮连续进行 10 次翻页。
- **预期结果**：每页图片的 CDN 加载渲染延时 **≤2.0s**，翻页无卡顿，无页面错位。
- **验收标准**：A2（加载流畅） | 优先级：P0

#### TC-003：中文 TTS 音频合成与播发
- **用例目的**：验证点击朗读响应时间。
- **测试步骤**：进入第 3 页，点击“朗读此页”按钮，记录至声音开始发出的延迟。
- **预期结果**：系统于 **≤1.0s** 内开始播发普通话标准语音，若再次点击同一页，直接从 OSS 缓存拉取。
- **验收标准**：A3（TTS 响应 ≤1.0s） | 优先级：P0

#### TC-004：录音跟读采集与管理
- **用例目的**：验证 MediaRecorder 的音频录制、上传、播放与彻底删除。
- **测试步骤**：
  1. 点击“录音跟读”，允许麦克风授权并朗读 10 秒；
  2. 点击“结束”，检查“我的录音”列表中是否自动增加 WebM/Opus 条目；
  3. 在线点击播放回听，测试下载本地及点击删除。
- **预期结果**：全链路闭环，删除后 OSS 对应 bucket 下的文件彻底物理移除。
- **验收标准**：A4（录音闭环） | 优先级：P0

#### TC-005：教师上课视频录像与回看
- **用例目的**：验证 `getDisplayMedia` 录课及视频流上传。
- **测试步骤**：教师点击“屏幕录屏”上课，课后点击“结束并保存”，学生进入日历及录播卡片查看该日期录屏。
- **预期结果**：异步视频转存上传正常，学生端可跨端流畅回看。
- **验收标准**：A5（录课回看可用） | 优先级：P1

#### TC-006：摄像头占用与权限释放 Bug 校验
- **用例目的**：验证退出直播间后硬件设备占用的释放。
- **测试步骤**：教师开启摄像头，随后点击关闭，或者直接退出直播视图，观察摄像头硬件指示灯。
- **预期结果**：系统成功调用 `track.stop()` 并置空 `MediaStream` 实例，指示灯瞬间熄灭，彻底释放设备占用，**规避了退出仍占用的 Bug**。
- **验收标准**：安全硬件释放 | 优先级：P0

#### TC-007：三语界面无缝实时热切换
- **用例目的**：验证 i18n 语境适配完整性。
- **测试步骤**：在 LanguageContext 菜单中任意在中文、俄语、哈萨克语进行三语热切换。
- **预期结果**：全局 UI 元素瞬间响应（**≤200ms**），无遗漏残存的不完整翻译。
- **验收标准**：A6（三语正常） | 优先级：P1

#### TC-008：Excel 大纲题目同步作业测试
- **用例目的**：验证导入大纲后题目的自动分配绑定。
- **测试步骤**：上传题库 excel 文件，学生进入对应课时页面查看练习区。
- **预期结果**：选择题及跟读句子自动关联至当前页，作答提交正常。
- **验收标准**：练习绑定 | 优先级：P1

#### TC-010：直播同步（WebSocket 指令）
- **用例目的**：验证教师端翻页与 Canvas 绘画的低延迟广播。
- **测试步骤**：教师在一端点击翻页或在 Canvas 涂鸦，观察学生端表现。
- **预期结果**：端到端同步延迟 **≤500ms**。
- **验收标准**：直播同步 | 优先级：P1

## 3. 安全加固清检与应对清单

针对当前的测试阶段，发布如下**安全加固三条红线清单**，必须在正式试点前完成处置：

1. **红线一：外部暴破防御 (隐藏端口)**
   - **风险**：当前服务器除了 `80` 端口外，还将 `3001` (backend 端口) 和 `5432` (PostgreSQL 默认端口) 直接向公网暴露，极易遭受注入和 SQL 密码破解攻击。
   - **加固对策**：在腾讯云 Lighthouse 的防火墙安全组中，**删除 3001 和 5432 的公网规则**，使其仅在 Docker Bridge 内部网络 (`lingo-net`) 中相互寻址通信。公网仅开放 `80` (HTTP) 与 `443` (HTTPS) 并通过 Nginx 进行反向代理与统一流量审查。
2. **红线二：默认密码强加固**
   - **风险**：数据库 PostgreSQL 默认用户名和密码采用了开发简易默认值，风险极大。
   - **加固对策**：在 `.env` 中更换为 16 位以上的大小写数字强混淆密码，重新 `docker-compose up -d` 重新构建实例。
3. **红线三：JWT Demo 密钥替换**
   - **风险**：JWT_SECRET 采用 Demo 值，极易导致 Token 伪造越权。
   - **加固对策**：生产环境的环境变量强行注入高强度的随机 Base64 字符串密钥，保障令牌防篡改。
