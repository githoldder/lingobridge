# Vibe Coding 与工程化管控 SOP

本模块规定了 AI 辅助编程或写作时的工程化约束，旨在确保任务有序推进、防范 AI 幻觉引发的错误雪球效应，并将 Token 消耗降至最低。

## 1. 核心状态机机制 (Ralph Mechanism)
AI 连续处理大任务会导致上下文窗口污染和注意力分散。必须采用**短周期循环**。

- **三件套防线**：
  1. `prds/prd.json` / `prds/prd.md`：极度细化的原子任务清单（解决"下一步干什么"）。
  2. `context/context.txt`：踩坑与经验日志（解决"不要重复犯错"，≤1000行压缩至 `context/memory.md`）。
  3. `Git Commit`：阶段性存档点（解决"写崩了怎么办"）。
- **执行闭环**：`读取 prds/ -> 执行单一原子任务 -> 测试验证 -> 成功后 Git 存档并更新 context/context.txt -> 结束本轮会话`。**每一轮必须是清醒的全新上下文**。

## 2. 极致节约 Token (流水线固化)
拒绝让 AI 反复进行相同的推理，实现任务边际成本趋近于 0。

- **核心思想**：AI 的价值在于“生成架构”，不在于“重复执行”。
- **判断标准**：凡是执行频率 ≥3 次的任务（如抓取数据、自动编译、测试回归），必须：
  1. 第一次让 AI 写出 Bash 脚本或固定格式的 Skill 模板。
  2. 将其固化到 `scripts/` 或 `skills/` 目录中。
  3. 后续遇到相同任务，**仅通过运行脚本执行，Token 消耗降为 0**。

## 3. Git 版本安全网 (核心避坑指南)
永远不要相信 AI 能够一次性完美修改全局文件。项目必须被单线程的节点备份和多分支的隔离管控保护起来。

- **初始化本地仓库**：项目开发的第一步，必须保证根目录存在 `.git` 本地仓库。如果没有，让 AI 先执行初始化，并做第一次全量存档。
- **日常节点存档 (Commit)**：AI 每次完成一个小节、单一段落的扩写或排版的调整后，必须立即进行 `git commit -m "具体的修改说明"`。做到“每步皆有存档”，确保随时可退回。
- **防御性切分支**：针对大段落重写或复杂新功能的增加，必须先建立分支 (`git checkout -b feature-xxx`)，在分支上沙盒编译并测试通过后，再合并回主线。
- **果断回滚 (Reset / Rollback)**：当 AI 修改代码引发编译报错，或其生成内容被用户判定为“未达预期/偏离方向”时，**绝对禁止**让 AI 在错误的上下文里继续“修修补补”从而引发错误滚雪球。应当果断要求 AI 执行 `git reset --hard` 等命令退回上一个稳定状态，丢弃当前错误上下文，换一种思路重新执行。

## 4. 文件夹与上下文管理 (PARA + 就近原则)
良好的目录结构能极大降低 AI 的检索难度。

- **命名规范**：推荐使用数字前缀+下划线，如 `01_Data/`, `02_Notes/`。
- **插图管理**：本项目采用单文件架构，所有插图统一存放在 `figures/` 目录。文件命名应语义清晰（如 `system_architecture.png`、`experiment_result.png`）。
- **上下文协议**：任务定义存 `prds/`，执行记录统一写入 `context/context.txt`；超过 1000 行压缩至 `context/memory.md`。
