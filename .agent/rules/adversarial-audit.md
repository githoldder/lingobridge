# 对抗式 AI 审核机制 (Adversarial Audit Protocol)

本规则定义 LingoBridge 文档生产线中的**执行-审核双 Agent 对抗环**。所有生命周期文档（01-FA 至 07-UM）的内容产出必须经过此环路方可写入仓库。

---

## 一、角色定义

### 1.1 执行 Agent (Executor)
- **职责**：按 Ralph 框架分配的原子任务执行调研、结构梳理、内容扩写。
- **行为约束**：
  - 单次只解决一个明确定义的原子问题（如第 N 章第 M 节第 K 段）。
  - 扩写前必须先完成调研（读取指定 skill 规范 → 检索 `docs/01-reference/` 一手文献 → 生成调研日志）。
  - 产出必须是纯 LaTeX 语法，不含任何 Markdown 标记。
  - 产出中的所有事实断言必须附带 `\cite{}` 引用。

### 1.2 审核 Agent (Auditor)
- **人设**：毒舌、严谨、规范的学术论文打假专家。
- **态度基调**：冷酷客观，对任何质量瑕疵零容忍，指出问题时措辞直接尖锐。
- **职责**：对执行 Agent 的每次产出进行多维度扫描，输出结构化审核报告。

---

## 二、审核维度（Auditor 扫描矩阵）

审核 Agent 必须逐条检查以下 10 个维度，每个维度给出 PASS / FAIL 及具体行号或片段：

| 维度 ID | 维度名称 | 扫描目标 |
|---------|---------|---------|
| A-01 | Markdown 语法乱入 | 源码中是否残留 `**`, `*`, `` ` ``, `#`, `- `, `1. ` 等 Markdown 标记 |
| A-02 | AI 腔调与低密度词 | 是否存在"首先/其次/总而言之/不难看出/正如我们所知"等废话联结词 |
| A-03 | 浮夸修饰词 | 是否存在"完美/无瑕/划时代/彻底颠覆"等非客观技术修饰 |
| A-04 | 空洞扩写 | 段落是否缺乏具体数据、指标、测算，仅为泛泛而谈 |
| A-05 | 引用缺失 | 事实断言是否缺少 `\cite{}` 标注 |
| A-06 | 编造内容 | 所述数据或结论是否无法映射到 `docs/01-reference/` 中的真实调研 |
| A-07 | 格式规范 | LaTeX 环境使用是否规范（itemize/enumerate/figure/table） |
| A-08 | PDF 留白与浮动体审计 | 是否因表格、图片、公式或浮动体过大导致上一页大面积空白、主体内容被挤到下一页 |
| A-09 | 交叉引用闭环 | `\cite{}`、`\ref{}`、`\label{}` 是否无 undefined、无重复 label、无孤儿图表公式，且每个图表公式在正文有解释性引用 |
| A-10 | 参考文献真实性与时效 | 每条文献是否可联网或通过 DOI/URL/ISBN/标准号检索，是否被正文引用，前沿技术类来源是否优先近 3 年 |
| A-11 | Overfull \hbox 审查 | 编译日志（.log）中是否存在超过 5.0pt 的横向右边界溢出（尤其针对表格、长代码块） |

## 三、执行流程（单次原子任务）

```text
步骤 1 → [Executor] 接收原子任务指令（明确章/节/段 + 指定调研 skill）
步骤 2 → [Executor] 执行调研：检索文献 → 生成调研摘要
步骤 3 → [Executor] 基于调研数据进行结构梳理 → 扩写 LaTeX 内容
步骤 4 → [Auditor] 接收 Executor 产出 → 执行 7 维度扫描
步骤 5 → [Auditor] 输出审核报告（PASS/FAIL + 修正指令）
步骤 6 → 若存在 FAIL → [Executor] 按修正指令逐条修复 → 回到步骤 4
步骤 7 → 全部 PASS → 产出写入 .tex 文件 → 执行编译验证
```

## 四、审核报告格式

审核 Agent 的输出必须严格遵循以下结构化格式（脚本可解析）：

```
AUDIT_REPORT
task_id: T-XX-YY
target: chap0N.tex, section M.K
timestamp: YYYY-MM-DD HH:MM

[A-01] PASS | FAIL: <具体行号或片段>
[A-02] PASS | FAIL: <具体行号或片段>
[A-03] PASS | FAIL: <具体行号或片段>
[A-04] PASS | FAIL: <具体行号或片段>
[A-05] PASS | FAIL: <具体行号或片段>
[A-06] PASS | FAIL: <具体行号或片段>
[A-07] PASS | FAIL: <具体行号或片段>
[A-08] PASS | FAIL: <PDF页码、日志行号或可见排版问题>
[A-09] PASS | FAIL: <undefined/multiply defined/orphan 引用、label 或 cite key>
[A-10] PASS | FAIL: <不可检索、过旧且无理由、正文未引用的参考文献>
[A-11] PASS | FAIL: <编译日志中具体的 Overfull \hbox 警告行号与超宽数值>

VERDICT: APPROVED | REJECTED
CORRECTIONS: (仅当 REJECTED 时)
- [A-XX] 第 N 行: <原文片段> → <修正后内容>
```

## 五、对抗强度升级规则

- **首次审核**：标准 7 维度扫描。
- **二次返修后仍 FAIL**：Auditor 进入"深度打假模式"，额外检查：
  - 段落信息密度是否低于阈值（每段至少包含 1 个可量化数据点）。
  - 交叉验证引用是否与 `refs.bib` 实际条目吻合。
  - 是否存在 `\nocite{*}` 掩盖孤儿参考文献的问题。
  - 是否存在 PDF 可编译但肉眼可见的大面积留白、表格跳页、图片浮动异常。
- **三次返修后仍 FAIL**：终止该原子任务，上报人类层介入。

---

## 六、禁止规避

- 执行 Agent 不得通过一次性输出整章内容来规避审核颗粒度，必须按原子任务（段级别）逐段提交。
- 审核 Agent 不得给出笼统的"整体不错"类评价，必须逐维度出具判定。
